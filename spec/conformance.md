# ACT Conformance

Status: Normative, companion to `spec/ACT-1.0.md` §16.5.

## 1. Profiles

| Profile | Requires |
| --- | --- |
| **Core** | Correct unsigned-event and artifact-type schema validation; canonicalization per RFC 8785; correct identifier derivation (§3 of ACT-1.0.md); acyclic lineage; append-only projections rebuildable from events alone. |
| **Cryptographic Integrity** | Core, plus: Ed25519 signing/verification; DSSE envelope construction/parsing; the six independent verification results of §4.5; key lifecycle (§11.2) enforcement; receipt hash-chain construction and tamper detection. |
| **Secure Service** | Cryptographic Integrity, plus: authentication (OIDC/JWT and/or Ed25519 service identity) kept separate from authorization; tenant isolation; policy-based authorization on every protected operation; secure defaults from `docs/threat-model.md`'s mitigation list; fail-closed production configuration. |
| **Federation** | Cryptographic Integrity, plus: bundle export/import (`spec/federation.md`); duplicate detection; partial-history representation; quarantine of invalid/untrusted imports; fork and equivocation detection. |
| **SDK** | Identical canonical bytes and identifiers to every other conformant SDK for the shared vector suite (`conformance/vectors/`); verification of signatures produced by any other conformant SDK; rejection of the shared negative-fixture suite. |
| **Explorer** | Secure Service, plus: every workflow in `PROMPT.md`'s ACT Explorer section is operable and passes the accessibility/responsive/visual checks in `docs/testing-strategy.md`. |

A repository or implementation claims a profile only when it passes every fixture category that profile lists below. The reference implementation in this repository is scoped per `docs/roadmap.md`; its current conformance claim is recorded in `conformance/CONFORMANCE_REPORT.md`, generated by `conformance/run-conformance.ts` — it is not asserted from prose.

## 2. Fixture Categories

- `conformance/fixtures/canonicalization/` — RFC 8785 canonicalization vectors: input JSON (varied key order, unicode, numeric edge cases) and expected canonical bytes/digest.
- `conformance/fixtures/signatures/` — Ed25519 sign/verify vectors: known keypair, known message, expected signature, and negative vectors (flipped bit, wrong key, truncated signature).
- `conformance/fixtures/schema/` — positive fixtures (must validate) and negative fixtures (must be rejected, with the specific violated constraint) for every schema under `schemas/`.
- `conformance/fixtures/state-machines/` — one fixture per transition (including invalid/forbidden transitions) per state machine in `spec/state-machines.md`.
- `conformance/fixtures/graph/` — DAG fixtures: valid multi-input/output transformations, valid merges, and cycle-introducing fixtures that MUST be rejected.
- `conformance/fixtures/approval-policy/` — quorum, expiry, revocation, supersession, and separation-of-duties fixtures.
- `conformance/fixtures/federation/` — export/import round trip, fork, duplicate, and partial-history fixtures.
- `conformance/fixtures/privacy/` — redaction and cryptographic-erasure fixtures, including expected post-redaction availability state.

## 3. Conformance Report

`conformance/run-conformance.ts` executes every fixture category against the SDKs present in this repository (TypeScript and Python today; Go and Rust runners do not exist yet, see `docs/roadmap.md`) and emits `conformance/CONFORMANCE_REPORT.json`, a machine-readable pass/fail list keyed by fixture ID, plus a human-readable Markdown summary. The SDK profile's cross-verification requirement is checked bidirectionally: `conformance/checks/sdk-interop.ts` calls `@act/crypto` in-process against a signature `sdks/python` produced, and shells out to a real Python process (`conformance/interop/verify-typescript-signed.py`) to check the reverse direction. `make verify` runs this report and fails if any fixture in a profile this repository claims does not pass.

## 4. Compatibility Rules Under Conformance

A conformant implementation:

- MUST negotiate `protocol_version` and reject an incompatible major version rather than guess (`ACT-1.0.md` §16.2).
- MUST preserve unknown-but-well-formed fields through a read-write round trip (schema evolution, §16.2-16.3).
- MUST NOT treat an unrecognized enum value or event type as equivalent to any known value (§16.4).
- MUST register new algorithms and extension namespaces through the process in `GOVERNANCE.md` before shipping vectors that depend on them.
